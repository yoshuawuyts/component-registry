# Design Guidelines

## Design Context

### Users

WebAssembly developers building with Components and WIT interfaces. They use
`wasm-cli` from the terminal to install, publish, and run Wasm packages — and
browse the web registry to discover what's available. Their context is
development workflow: fast iteration, dependency management, and composing
components. The job to be done is **finding, installing, and shipping Wasm
packages with zero friction**.

### Brand Personality

**Minimal, Clean, Professional.**

The tone is direct and technical — like well-written documentation. No marketing
speak, no unnecessary flair. Information density is preferred over decoration.
The tool should feel like a sharp, reliable instrument that stays out of the way.

### Aesthetic Direction

- **Visual tone**: Technical minimalism. Generous whitespace, clear typography
  hierarchy, restrained color use.
- **Reference**: [zed.dev/extensions](https://zed.dev/extensions) — clean grid
  of packages, search-first, fast-loading, no noise.
- **Anti-reference**: Flashy startup landing pages — style over substance,
  gratuitous animation, marketing-heavy layouts.
- **Emotional goal**: Speed — "this feels fast and efficient."
- **Theme**: Light theme by default, dark theme via `prefers-color-scheme`.
  Standard terminal palette for CLI/TUI.

### Color System

All frontend colors use OKLCH, rooted in the Wasm logo purple (hue 280).
Neutrals use hue 290 for a violet tint. All text tokens pass WCAG AA (4.5:1)
against `bg`. See `layout.rs` for the full token list.

| Token          | Light                     | Dark                      | Usage                                        |
|----------------|---------------------------|---------------------------|----------------------------------------------|
| Accent         | `oklch(0.49 0.257 280)`   | `oklch(0.70 0.16 280)`    | Primary interactive color, links, highlights |
| Text primary   | `oklch(0.20 0.03 290)`    | `oklch(0.94 0.01 290)`    | Body text, headings                          |
| Text secondary | `oklch(0.40 0.03 290)`    | `oklch(0.78 0.025 290)`   | Descriptions, metadata                       |
| Text muted     | `oklch(0.54 0.025 290)`   | `oklch(0.66 0.03 290)`    | Timestamps, help text                        |
| Text faint     | `oklch(0.56 0.02 290)`    | `oklch(0.62 0.025 290)`   | Version numbers, counts, minor labels        |
| Border         | `oklch(0.91 0.018 290)`   | `oklch(0.32 0.04 290)`    | Card borders, dividers                       |
| Background     | `oklch(1 0 290)`          | `oklch(0.185 0.025 290)`  | Page background                              |
| Surface        | `oklch(0.975 0.006 290)`  | `oklch(0.23 0.03 290)`    | Inline code backgrounds, hover surfaces      |

**CLI colors**: Green (success/checkmarks), Yellow (active/progress bars),
Dim (secondary info like versions and sizes).

**TUI colors**: Yellow (active state, headers), Cyan (titles, counts),
Dark Gray (help text, disabled), Magenta/Blue/Green (WIT syntax highlighting).

### Typography

- System font stack for body text (Tailwind defaults).
- Monospace for package names, code references, versions, and WIT identifiers.
- Size scale: `text-sm` (metadata), `text-lg`/`text-xl` (section headings),
  `text-3xl` (page headings), `text-6xl` (error states like 404).

### Design Principles

1. **Content over chrome** — Every pixel should serve a purpose. No decorative 
   elements that don't aid comprehension. Package data is the interface.

2. **Speed is a feature** — The UI should feel instant. Server-side rendering,
   minimal JavaScript, aggressive caching, progress indicators that show real
   work happening. No loading spinners hiding slow backends.

3. **Terminal-native** — The CLI and TUI are first-class interfaces, not
   afterthoughts. Respect `NO_COLOR`, use standard terminal conventions, and
   prefer information density over visual embellishment.

4. **Consistency across surfaces** — The same package should feel recognizable
   whether viewed in the CLI, TUI, or web frontend. Use the same naming
   conventions (`namespace:name`), the same data hierarchy, and the same
   accent color where applicable.

5. **Accessible by default** — Standard accessibility best practices: sufficient
   color contrast, semantic HTML, keyboard navigation, respect for user
   preferences (color, motion).
